Stage 2 · date-only features

The deterministic backbone

Stage 2 fits a model whose features depend only on the date: a trend (slow drift, Stage 0 confirmed) plus Fourier seasonal terms (weekly + annual, justified by the periodogram and the month profile). "Deterministic" means computable for any future date — perfect for forecasting, because we always know the date.

Source: analyze/concepts/seasonality-fourier.md, analyze/concepts/lag-horizon.md.

Stage 2 · trend + seasonality

Why sin + cos represent a cycle

how it works Two columns, any peak position

A cycle is a wave; the natural way to write one is sin and cos. For a weekly pattern (period = 7): sin(2π·day/7) and cos(2π·day/7) each complete one full wave every 7 days. Why both? A single sine has a fixed peak position; combining a sin and a cos (with weights the model learns) lets it place the peak anywhere, at any height — so with just 2 numbers per cycle a plain linear regression fits "this series peaks Saturday, that one Sunday."

The model learns two weights (w_sin, w_cos) plus an intercept so intercept + w_sin·sin + w_cos·cos traces this shape. Two columns reconstruct the weekly rhythm and extend cleanly into any future date. (A sharp single-day dip like Sunday's needs a couple more Fourier pairs — the higher harmonics — to pin down exactly. That's why the periodogram's 3.5-day spike matters — see Stage 0 · periodogram.)

tooling CalendarFourier & DeterministicProcess

CalendarFourier generates those sin/cos columns from the dates. Instead of 365 separate "is it Jan 1 / Jan 2 / …" columns for the yearly cycle, you ask for a few Fourier pairs and get a handful of smooth columns approximating the whole annual wave. Stage 0 told us how many: ~2–3 weekly and ~3–5 annual harmonics.

DeterministicProcess assembles the full time-feature table: a constant, a trend (1, 2, 3, … counting upward), and the seasonal Fourier terms. We allow mild curvature for the trend because Stage 0 showed growth is decelerating.

why gaps wreck it "Out of phase" — in numbers

Phase = where you are in the cycle. The sin/cos value comes from the row's position, assuming consecutive days. Correct (no gap):

datereal weekdayposition dsin usedsales
Aug 12Sat5−0.9751630
Aug 13Sun6−0.782952

Now drop the Sat Aug 12 row (a closed day). Everything after shifts up one:

datereal weekdayposition d (miscounted)sin usedsales
Aug 13Sun5 (thinks it's Sat)−0.975952

Sunday's 952 sales are now paired with Saturday's sin value. Every weekday past the gap is mislabelled by one — silent corruption. This is exactly why Trap 1's reindex must run before any seasonal feature.

Punchline

Fill the calendar first (so position = real date again), then build Fourier seasonality.

Stage 2 · the lag floor

The 16-day lag horizon (why lags must be ≥ 16)

The deterministic backbone uses only date features. Once we add lag features in Stage 3 ("sales N days ago"), they're built from past sales. But our prediction window is 16 days long and we have no real sales inside it. That puts a hard floor on how short a lag we can use.

TRAIN (real sales known)            TEST / horizon (must PREDICT, no real sales)
... Jul 31 ... Aug 15  │  Aug 16  Aug 17  ...  Aug 30  Aug 31
            ↑ last real day        └──────── 16 days to predict ────────┘
the problem Why lag_7 breaks partway through

Check lag_7 for each test day — does "7 days before" land in the known zone (≤ Aug 15)?

Predict this day7 days beforeKnown?
Aug 16Aug 9✅ in train
Aug 22Aug 15✅ last real day
Aug 23Aug 16❌ a test day — no real sales
Aug 31Aug 24❌ test day

From Aug 23 onward, "7 days ago" points back into the test period itself.

the fix Why lag_16 works for the whole horizon
Predict this day16 days beforeKnown?
Aug 16Jul 31✅ in train
Aug 31Aug 15✅ last real day

Even the farthest day (Aug 31) reaches back exactly to Aug 15.

The rule

The last day you predict is 16 days past your last real data. A lag that must work for the whole horizon has to reach back at least 16 days. In general: minimum usable lag = horizon length (H).

The design decision: direct forecasting with base lag = 16

ApproachHowTrade-off
Direct (chosen)Predict all 16 days at once, using only real lags ≥ 16 (lag_16, lag_17, … ; rolling windows anchored at shift(16))One feature matrix, no error compounding
Recursive (not chosen)Predict Aug 16, treat it as real, use as lag_1 for Aug 17, …Allows short lags, but errors snowball across 16 days
Direct per stepA separate model per horizon day (day-1 uses lag_1, day-16 uses lag_16)Freshest legal lag per day, but 16 models
Long horizons → lean on seasonality

A 1-year horizon would need lag_365 as its shortest lag — nearly useless. The longer the horizon, the less lags help and the more you lean on date-only features (trend + Fourier seasonality, holidays), computable for any future date. That's why the modelling is staged: a deterministic backbone first (this Stage), lags added on top as a short-horizon booster (Stage 3) — never a replacement.
