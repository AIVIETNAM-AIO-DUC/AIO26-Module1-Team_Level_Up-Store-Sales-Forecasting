Foundations · what we predict

What are we actually predicting?

A supermarket chain in Ecuador wants to predict daily sales. There are 54 stores, each selling 33 product families (BEVERAGES, PRODUCE, AUTOMOTIVE, …), recorded once per day.

Given years of history (2013-01-01 → 2017-08-15), predict sales for the next 16 days (2017-08-16 → 2017-08-31) for every store + family. That fixed window has a name we reuse everywhere: the horizon (= 16 days).

Key word · series

A series is the day-by-day sales history of one product family at one store — the "thing that has a past and a future." Almost every operation happens per series, never mixing across them.

54 stores × 33 families = 1,782 series

Foundations · the metric · interactive

The metric: RMSLE

Read the name backwards and it's a recipe, one word per step:

WordStep
Errorhow wrong each prediction is: prediction − actual
Logarithmicbut first take the log of both numbers
Squaredsquare each error (too-high and too-low both count)
Meanaverage all those squared errors
Roottake the square root at the end
 
RMSLE = sqrt( mean( (log1p(prediction) − log1p(actual))² ) )
Why the log?

It measures relative (ratio) error, not the raw gap. Predict 15 when truth is 5, and predict 3,000 when truth is 1,000 — both are "3× too high," equally bad. Because log(a) − log(b) = log(a/b), RMSLE asks "how many times off were you?" That's how a tiny family and a huge family count fairly.

Unpack — why this matters

Product families in this dataset live on wildly different scales: GROCERY I sells a few thousand units/day, while BABY CARE sells only a handful. With plain RMSE (which measures "raw gap"), the same proportional miss is punished very differently:

FamilyPredictedActualRaw gapRatio
GROCERY3,0001,0002,0003×
BABY CARE155103×

RMSE punishes GROCERY 200× more than BABY CARE — even though both are "3× too high." The model learns to pour effort into the big family and abandon the small one.

How log fixes it. The key identity: log(a) − log(b) = log(a/b). Try both pairs:

  • log(3000) − log(1000) = log(3000/1000) = log(3)
  • log(15) − log(5) = log(15/5) = log(3)

Both pairs return the same value log(3) — once you take the log, "distance" stops depending on scale and depends only on the ratio. So RMSLE punishes both pairs equally.

One-line summary: log converts the number line from "gap scale" to "ratio scale" → small-family and big-family forecasts get scored fairly.

log1p(x) = log(1+x) so log1p(0)=0 (no crash on zero sales); its inverse expm1 returns real units. Two consequences we apply to every model: we train in log space, and we clip predictions to ≥ 0.

Foundations · the scoreboard

How do we know a model is good? An honest holdout

A model always looks brilliant on the data it trained on — it has effectively seen the answers. The only honest signal is its error on data it did not train on. We keep a private slice of our own data as an instant, unlimited scoreboard: a holdout.

Time-series rule

Random K-fold cheats here. Validate on a day in the middle of history and the model trains on days that come after it — it sees the future relative to what it predicts. The split must be a clean cut in time.

train: 2013-01-01 ........... 2017-07-30  │  holdout: 2017-07-31 → 2017-08-15
       (model learns from the past)        │  (model predicts forward, blind)

We hold out the last 16 training days — same length, same "predict 16 days forward" shape as the real task.

The holdout is NOT test.csv — keep them straight

Foundations · vocabulary

Five terms the rules use

The rules below assume five words. Each is a general time-series concept that applies far beyond this project; the example is drawn from this dataset so it stays concrete.

  • Leakage. Using information at training time that wouldn't be available when the model is asked to predict — the model "cheats" by seeing the future. Example: training on Aug 16 sales to predict Aug 17 would be leakage; Aug 16 isn't known when we submit on Aug 15.
  • Lag. A past value of the series brought onto the present row as a feature. lag_7 on today's row = the value from 7 days ago in the same series. Example: df["sales"].shift(7) computed per (store, family).
  • Trend. A slow, persistent direction of change over long stretches (months or years). Example: across 2013 → 2017 this chain's overall sales drift upward — the long-run slope, ignoring weekly wiggles.
  • Seasonality. Repeating cycles tied to calendar position — weekly (each weekday sells differently), yearly (December peak), etc. Usually modeled with Fourier sin/cos terms or day-of-week one-hots. Example: every Sunday in this dataset sells more than every Tuesday.
  • Direct forecasting. Predicting all H horizon days in one shot (one model per horizon step, or a multi-output model) instead of predicting day 1, feeding it back, then predicting day 2 ("recursive"). With direct, no model output is ever used as an input — errors don't compound — but every lag must be ≥ H days old. In this project H = 16, so the smallest usable lag is 16.
Foundations · the guard rails

Five rules of honest time-series forecasting

Everything above distills into five rules. Each is a general principle that holds for any forecasting task on time-ordered data; the sidebar shows how this project enforces it in code.

  1. No leakage. Every feature used at training time must be knowable at prediction time. If a row about Aug 17 uses any value the world doesn't yet know on Aug 15, the model is cheating.
    In this project

    Oil prices are forward-filled (never interpolated — interpolation would pull in later values); transactions are lag-only (the same-day value isn't known when we predict); src/validation.py :: assert_no_leak() raises if any feature peeks forward.

  2. Gap-free before features. Time-series features count by row position, not by date. Reindex every series onto a complete calendar (daily, hourly, whatever your grain is) before generating any lag or seasonal feature, otherwise "7 rows back" silently stops meaning "7 days back."
    In this project

    Stores close on Dec 25 → missing rows. reindex_series_gapfree() inserts them as sales = 0 with was_closed = True; assert_gapfree() raises if any gap remains. Full walkthrough: Trap 1.

  3. Lags must clear the horizon. Under direct forecasting, every lag fed into the model must be at least H steps old, where H is the horizon length. A shorter lag wouldn't be knowable on the day we predict the far edge of H.
    In this project

    H = 16, so base_lag = 16. The last day we predict (Aug 31) is 16 days past our last real data (Aug 15); a lag-7 feature would be missing from Aug 22 onward. See the lag-horizon deep dive.

  4. Time-respecting validation. Validation data must be strictly later in time than training data. Random K-fold splits across time and lets the model train on the future of the days it's about to "predict."
    In this project

    One 16-day holdout (Jul 31 → Aug 15) mirrors the real horizon; train_holdout_split() performs the cut and asserts the ordering.

  5. Consistent metric handling. Compute the metric identically across every model and every prediction — same transform, same clipping, same averaging — or scores stop being comparable.
    In this project

    RMSLE on log1p, predictions clipped to ≥ 0 before scoring; src/validation.py :: rmsle() and clip_nonneg() are the only entry points.

Meta-lesson

When a rule matters, encode it as an assertion. A guard that raises the moment you break a rule is worth more than a comment reminding you not to.
