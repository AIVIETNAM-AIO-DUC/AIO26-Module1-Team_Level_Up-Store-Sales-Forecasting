Foundations · what we predict

What are we actually predicting?

A supermarket chain in Ecuador wants to predict daily sales. There are 54 stores, each selling 33 product families (BEVERAGES, PRODUCE, AUTOMOTIVE, …), recorded once per day.

Given years of history (2013-01-01 → 2017-08-15), predict sales for the next 16 days (2017-08-16 → 2017-08-31) for every store + family. That fixed window has a name we reuse everywhere: the horizon (= 16 days).

Key word · series

A series is the day-by-day sales history of one product family at one store — the "thing that has a past and a future." Almost every operation happens per series, never mixing across them.

54 stores × 33 families = 1,782 series

Foundations · the metric · interactive

The metric: RMSLE

Read the name backwards and it's a recipe, one word per step:

WordStep
Errorhow wrong each prediction is: prediction − actual
Logarithmicbut first take the log of both numbers
Squaredsquare each error (too-high and too-low both count)
Meanaverage all those squared errors
Roottake the square root at the end

As a formula — first the origin form, exactly as the competition defines it:

                   1    n
RMSLE  =  sqrt(   ───   Σ   ( ln(1 + pᵢ) − ln(1 + aᵢ) )²   )
                   n   i=1

   n  = number of predictions        pᵢ = the i-th prediction
   ln = natural logarithm            aᵢ = the i-th actual value

That's the recipe in symbols. For each pair — a prediction pᵢ and its matching actual aᵢ:

  1. Logarithmic — take the log of both numbers: ln(1 + pᵢ) and ln(1 + aᵢ).
  2. Error — subtract the two logs; that difference is the error, measured in log space.
  3. Squared — square that difference.
  4. Mean — do this for all n pairs and average the squares.
  5. Root — finally take the square root.

What log1p means, step by step

The code — and the shorthand formula below — writes log1p(x) instead of ln(1 + x). They are the same function; the name spells out "log of 1 plus x": log1p(x) = ln(1 + x).

x = 15 :  1 + x = 16  →  ln(16) ≈ 2.7726   →  log1p(15) ≈ 2.7726
x = 0  :  1 + x = 1   →  ln(1)  = 0        →  log1p(0)  = 0

That +1 is the whole trick: plain log(0) is undefined and would crash, but log1p(0) is a clean 0. (The why behind the +1, and its inverse expm1, are explained just below.)

With log1p defined, the origin formula compresses to the shorthand you'll meet in the code — same math, shorter to write:

RMSLE = sqrt( mean( (log1p(prediction) − log1p(actual))² ) )
Why the log?

Taking the log before measuring error makes the metric care about the ratio of the miss ("how many times off?") instead of the raw gap between two numbers. The "Unpack" card below makes that concrete with an example.

Unpack — why this matters

Product families in this dataset live on wildly different scales: GROCERY I sells a few thousand units/day, while BABY CARE sells only a handful. With plain RMSE (which measures "raw gap"), the same proportional miss is punished very differently:

FamilyPredictedActualRaw gapRatio
GROCERY3,0001,0002,0003×
BABY CARE155103×

RMSE punishes GROCERY 200× more than BABY CARE — even though both are "3× too high." The model learns to pour effort into the big family and abandon the small one.

How log fixes it. The key identity: log(a) − log(b) = log(a/b). Try both pairs:

  • log(3000) − log(1000) = log(3000/1000) = log(3)
  • log(15) − log(5) = log(15/5) = log(3)

Both pairs return the same value log(3) — once you take the log, "distance" stops depending on scale and depends only on the ratio. So RMSLE punishes both pairs equally.

One-line summary: log converts the number line from "gap scale" to "ratio scale" → small-family and big-family forecasts get scored fairly.

Why a dedicated log1p function?

log1p(x) = log(1 + x) is true by definition (1p = "1 plus"). So why ship a separate log1p instead of just writing log(1 + x)? Two reasons:

1 — it dodges log(0). Sales are often 0 (closed days, slow products), and plain log(0) is undefined (it heads to −∞ and crashes the metric). Adding 1 first moves that danger point away: log1p(0) = log(1) = 0.

2 — it stays accurate when x is tiny. A computer keeps only ~15–16 digits, so the naive 1 + x can silently lose a very small x (e.g. 1 + 1e-16 rounds back to 1.0, so log returns 0 — wrong). log1p uses a series expansion for small x and stays correct; its inverse expm1(x) = eˣ − 1 (which returns real units) exists for the same precision reason.

Two consequences we apply to every model: we train in log space, and we clip predictions to ≥ 0.

Foundations · the scoreboard

How do we know a model is good? An honest validation set

A model always looks brilliant on the data it trained on — it has effectively seen the answers. The only honest signal is its error on data it did not train on. We keep a private slice of our own data as an instant, unlimited scoreboard: a validation set.

Time-series rule

Random K-fold cheats here. Validate on a day in the middle of history and the model trains on days that come after it — it sees the future relative to what it predicts. The split must be a clean cut in time.

train: 2013-01-01 ........... 2017-07-30  │  validation: 2017-07-31 → 2017-08-15
       (model learns from the past)        │  (model predicts forward, blind)

We hold out the last 16 training days — same length, same "predict 16 days forward" shape as the real task.

The validation set is NOT test.csv — keep them straight

Foundations · the guard rails

Five rules of honest time-series forecasting

Five rules sit under every modeling decision in this project. Each is a general principle that holds for any forecasting task on time-ordered data; each one below is grounded in our 2013-01-01 → 2017-08-15 daily sales data and the 16-day Aug 16 → Aug 31 horizon, and points to the code that enforces it.

Concrete scenario — what rule 1 & rule 3 look like in this dataset

In the raw Kaggle files we can't see the future: test.csv (Aug 16 → Aug 31) has no sales column. Leakage only becomes a real risk when we cut our own validation set during development — post-cutoff sales still sit in train.csv and can be accidentally pulled in by sloppy feature code (rule 1).

At submission time we predict all 16 days (Aug 16 → Aug 31) from data ending Aug 15. The same lag_k feature has to work on every day in the horizon (rule 3):

  • lag_2 for Aug 18 needs sales from Aug 16 → not yet known ❌
  • lag_16 for Aug 31 needs sales from Aug 15 → available ✅

The smallest lag that works for all 16 horizon days = 16. That's why base_lag = 16.

  1. No leakage. Leakage means a feature on a training row uses information the world wouldn't yet have on the day that row would be predicted — the model "cheats" by seeing the future. Every feature must be knowable at the matching prediction day. In our data: oil.csv only quotes weekdays, so Sat/Sun oil prices are missing. Interpolating between Fri 2017-07-28 and Mon 2017-07-31 leaks Monday's price backward into Friday's row; forward-fill (carry Friday's price into Sat/Sun) never reaches into the future. transactions.csv is a harder version of the same trap: the file ends 2017-08-15 and isn't provided for the Aug 16 → Aug 31 horizon, so it's usable only as a lagged feature, never same-day. Guarded by src/validation.py :: assert_no_leak().
  2. Gap-free before features. A lag is a past value of the series carried onto the current row as a feature (lag_7 for today = the value seven rows back). Seasonality features — sin/cos of day-of-year, day-of-week one-hots, etc. — likewise assume consecutive rows. Both count by row position, not by date, so any missing day silently shifts every lag and breaks every seasonal alignment. In our data: every store closes on Dec 25, so those rows are absent from train.csv. Without reindexing, lag_7 on 2014-01-01 silently reaches 2013-12-24 (not Dec 25). reindex_series_gapfree() inserts the missing days with sales = 0, was_closed = True per (store, family); assert_gapfree() raises if any gap remains. Full walkthrough: Trap 1.
  3. Lags must clear the horizon. Under direct forecasting — one prediction per horizon day in a single shot, with no model output ever fed back as input — every lag must be at least H rows old, where H is the horizon length. A shorter lag wouldn't be knowable on the day we predict the far edge of H. In our data: H = 16, so base_lag = 16. The last day we predict (Aug 31) is 16 days past our last real data (Aug 15); lag_7 works Aug 16 → 22 but breaks from Aug 23 onward (Aug 23's lag_7 source = Aug 16, itself a test day with no real sales). See the lag-horizon deep dive.
  4. Time-respecting validation. Validation data must be strictly later in time than training data. Random K-fold cuts across time and lets the model train on the future of the days it's about to "predict," producing scores that don't survive the real test. In our data: the real test is Aug 16 → Aug 31, 2017 — 16 days right after training ends. The development validation set mirrors that exactly: train on 2013-01-01 → 2017-07-30, validate on 2017-07-31 → 2017-08-15 (same 16-day window, just shifted back 16 days). train_validation_split() performs the cut and asserts the ordering.
  5. Consistent metric handling. Compute the metric identically across every model and every prediction — same transform, same clipping, same averaging — or the scoreboard stops being comparable. In our data: Kaggle scores this competition as RMSLE on log1p(sales), so we do the same on every validation score. Sales can't be negative, but a regression model can still emit a small negative — an unclipped negative crashes log1p, so we clip predictions to ≥ 0 first. src/validation.py :: rmsle() and clip_nonneg() are the only entry points.
Meta-lesson

When a rule matters, encode it as an assertion. A guard that raises the moment you break a rule is worth more than a comment reminding you not to.
